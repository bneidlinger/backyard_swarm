Initial Setup & Environment Preparation

1System Requirements

Before you begin, ensure each machine meets these requirements:

  • GPU: NVIDIA RTX 3070, 4060, 5090, or similar
  • OS: Ubuntu 20.04+ or Windows with WSL2
  • RAM: 16GB+ recommended
  • Storage: 50GB+ free space
  • Network: Reliable internet connection

2Software Prerequisites

Install these components on all machines:

  • Docker with NVIDIA Container Runtime
  • NVIDIA GPU drivers (latest stable)
  • Python 3.10+ (for running jobs)
  • Git (for cloning repositories)

3Network Setup

Choose your networking approach:

  • LAN Only: All machines on same network
  • Tailscale (Recommended): Secure mesh VPN
  • WireGuard: Self-hosted VPN solution
  • Public Internet: Advanced configuration needed
Pro Tip: Start with a 2-machine setup on your local network before expanding. This lets you verify everything works before adding complexity.

Deployment Walkthrough

Step 1: Prepare Your Machines

Install Dependencies (Ubuntu/WSL) 
# Run on all GPU machines
cd backyard_swarm_scripts
sudo bash scripts/bs_install_linux.sh

# Join Tailscale network (recommended)
sudo tailscale up

# Verify installation
bash scripts/bs_detect.sh

Step 2: Build Docker Images

Build Images (Run Once) 
# Build coordinator and worker images
bash docker/build.sh

# Verify images were created
docker images | grep backyard

Step 3: Start the Coordinator

Choose Your Coordinator Machine

The coordinator should be:

  • Always available: Raspberry Pi, server, or dedicated machine
  • Stable network: Consistent IP address or hostname
  • Central location: Good connectivity to all workers
Start Coordinator 
# On coordinator machine
bash scripts/bs_coordinator_up.sh

# Check coordinator status
bash scripts/bs_status.sh
Success Indicators: You should see MinIO and coordinator containers running, and be able to access `http://coordinator-ip:8080/healthz` in your browser.

Step 4: Join GPU Workers

Start Workers (Each GPU Machine) 
# Replace with your coordinator's IP address
COORDINATOR_IP="192.168.1.100"  # or Tailscale IP like 100.x.y.z

# Start worker
bash scripts/bs_worker_up.sh -c http://${COORDINATOR_IP}:8080

# Verify worker is running
docker logs -f backyard-worker
Tailscale Users: Use your Tailscale IP addresses for better security and easier setup. Find them with `tailscale ip -4`.

Monitoring Your Swarm

Ray Dashboard

Primary monitoring interface for your cluster:

  • URL: http://coordinator:8265
  • Shows: Node topology, resource usage, active jobs
  • Real-time: Updates as jobs run and complete

API Endpoints

Programmatic access to cluster status:

  • /healthz - Service health check
  • /workers - List registered workers
  • /cluster/status - Detailed cluster info

Container Logs

Debug issues with detailed logging:

  • docker logs backyard-coordinator
  • docker logs backyard-worker
  • Tip: Use -f flag for live following

Health Check Commands

System Status Overview 
# Check overall system health
bash scripts/bs_status.sh http://coordinator:8080

# Get detailed worker information
curl http://coordinator:8080/workers | jq

# View cluster resources
curl http://coordinator:8080/cluster/status | jq
Important: Workers send heartbeats every 30 seconds. If a worker stops responding for 5 minutes, it's automatically removed from the cluster.

Writing Custom Jobs

Custom Ray Job Template 
import ray

# Connect to your swarm
ray.init(address="auto")

@ray.remote(num_gpus=0.5)  # Use half a GPU per task
def process_batch(data_batch):
    """Your custom processing logic here"""
    import torch
    device = torch.device("cuda" if torch.cuda.is_available() else "cpu")

    # Your GPU-accelerated code
    result = your_model(data_batch.to(device))
    return result.cpu()

# Submit work across the mesh
batches = [batch1, batch2, batch3, ...]
futures = [process_batch.remote(batch) for batch in batches]

# Collect results
results = ray.get(futures)
print(f"Processed {len(results)} batches across the swarm!")
Resource Management: Use num_gpus=0.5 to run multiple tasks per GPU, or num_gpus=1 for exclusive access. Ray automatically handles scheduling across your heterogeneous hardware.

Performance Optimization

GPU Memory Management

Optimize memory usage for your workloads:

  • Set PYTORCH_CUDA_ALLOC_CONF for memory efficiency
  • Use gradient checkpointing for large models
  • Monitor memory usage via Ray dashboard
  • Adjust batch sizes based on available VRAM

Network Optimization

Minimize network bottlenecks:

  • Use local storage for large datasets
  • Compress data before transmission
  • Consider data locality in task scheduling
  • Monitor bandwidth usage in production

Heterogeneous Hardware

Make the most of mixed GPU types:

  • Assign compute-heavy tasks to high-end GPUs
  • Use older GPUs for preprocessing/postprocessing
  • Balance workloads based on VRAM capacity
  • Set appropriate num_gpus values per task

Environment Tuning

Worker Performance Configuration 
# Create optimized worker configuration
cat > configs/production.env << EOF
# Memory optimization
PYTORCH_CUDA_ALLOC_CONF="max_split_size_mb:128,garbage_collection_threshold:0.8"

# GPU architecture targeting
TORCH_CUDA_ARCH_LIST="8.6;8.9"  # RTX 3070=8.6, RTX 4060=8.9

# Ray worker settings
RAY_memory_usage_threshold=0.95
RAY_object_store_memory_threshold=0.8

# Custom paths
MODELS_DIR=/fast_storage/models
DATA_DIR=/fast_storage/data
EOF

# Apply configuration
source configs/production.env

Troubleshooting Guide

üîç Worker Can't Connect

Symptoms: Worker logs show connection errors

  • Check network connectivity: ping coordinator-ip
  • Verify coordinator is running: curl coordinator:8080/healthz
  • Check firewall rules for ports 8080, 8265, 10001
  • Ensure Tailscale is connected on both machines

üö´ GPU Not Detected

Symptoms: Tasks run on CPU instead of GPU

  • Run bash scripts/bs_detect.sh to check GPU status
  • Verify NVIDIA drivers: nvidia-smi
  • Test Docker GPU access: docker run --gpus all nvidia/cuda:12.1-runtime-ubuntu20.04 nvidia-smi
  • Restart Docker daemon if needed

üíæ Out of Memory Errors

Symptoms: CUDA out of memory exceptions

  • Reduce batch sizes in your jobs
  • Lower num_gpus to allow memory sharing
  • Enable gradient checkpointing for models
  • Monitor VRAM usage in Ray dashboard

üêå Jobs Not Distributing

Symptoms: All work runs on one machine

  • Check worker registration: curl coordinator:8080/workers
  • Verify Ray cluster in dashboard: coordinator:8265
  • Ensure jobs use ray.init(address="auto")
  • Check resource requirements match available GPUs
